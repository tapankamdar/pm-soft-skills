# PM Writing & Documents

This skill covers the core written documents PMs produce every week: the status update that gets read, the PRD that gets approved, and the message to leadership that builds trust instead of creating noise.

## Core principle

All PMs write documents. Great PMs write documents that drive decisions — not just awareness. The test is not whether you wrote it. It's whether the reader knew what to do after reading it.

> "I have not read a status update you sent in the past 3 months. Until today." — GoDaddy CFO, after reading a 3-2-1 update

## Before responding — gather context with AskUserQuestion

Before diving in, ask the following questions using AskUserQuestion. Ask all questions in a single call (no more than 3).

**Question 1 — What are you trying to write?**
Options: `Status update or weekly report`, `PRD or feature spec`, `Message to my manager or leadership`, `Other`

**Question 2 — Who is the primary reader?**
Options: `My direct manager`, `Senior leadership or executives`, `My engineering or design team`, `Cross-functional stakeholders`

**Question 3 — What's the main challenge?**
Options: `Starting from scratch — I need a structure`, `I have a draft I want to improve`, `My reader doesn't engage with what I send`, `I'm not sure what to include`

Use the answers to select the right reference file and tailor the response.

## How to help

1. Ask the 3 context questions above using AskUserQuestion.
2. Read the reference file that matches the PM's task.
3. Apply the framework — including the specific template, rules, or script where one exists.
4. If the PM shares a draft, evaluate it against the framework and give specific, direct feedback.
